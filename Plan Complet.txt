Plan Complet et Détaillé de l'Application de Gestion Client et Emailing - VERSION FINALE
1. Vue d'ensemble du projet
Nom du projet : Application de gestion de clientèle et d'emailing pour petits commerçants
Objectif : Créer une application desktop multi-OS permettant aux petites entreprises locales de gérer leurs clients, leurs ventes, et leurs campagnes d'emailing de manière simple et professionnelle, sans connaissances informatiques requises.
Public cible : Petits commerçants (fleuristes, friperies, boutiques locales, etc.) avec une base de 1 000 à 10 000 clients (dimensionné jusqu'à 50 000 max)

2. Stack technique
Technologies principales :
	•	Langage : Python
	•	Interface graphique : PySide6 (Qt for Python)
	•	Base de données : SQLite (fichier local unique)
	•	Graphiques : Plotly (intégré via QWebEngineView)
	•	Emailing : smtplib + email (bibliothèques Python natives) + imaplib (réception emails)
	•	Service d'envoi automatisé : Daemon/Service système ou scheduler Python (APScheduler)
	•	Architecture : MVVM (Model-View-ViewModel)
Compatibilité :
	•	OS supportés : macOS, Windows, Linux
	•	Développement principal : macOS
	•	Tests : macOS + Windows
Environnement :
	•	Virtualenv : venv avec toutes les dépendances
	•	Gestion des dépendances : requirements.txt

3. Système de modes d'utilisation
Mode Vendeur (par défaut) :
Accès libre à :
	•	Onglet "Client" (création/modification)
	•	Onglet "Vente" (enregistrement des achats)
	•	Onglet "Recherche" (consultation des profils clients)
Accès bloqué à :
	•	Onglet "Emailing"
	•	Onglet "Boîte de réception"
	•	Onglet "Statistiques"
	•	Onglet "Mails enregistrés"
	•	Onglet "Historique"
	•	Onglet "Calendrier"
	•	Onglet "Produits"
	•	Onglet "Paramètres"
Mode Patron (protégé par mot de passe) :
Accès total à tous les onglets et fonctionnalités
Système de verrouillage :
	•	Icône cadenas visible en permanence dans l'interface
	•	Cadenas fermé = Mode vendeur
	•	Cadenas ouvert = Mode patron
	•	Clic sur le cadenas fermé : Demande du mot de passe → Déverrouillage
	•	Clic sur le cadenas ouvert : Verrouillage instantané (retour au mode vendeur)
Sécurité du mot de passe :
	•	Stockage : Hashé et sécurisé dans la base de données
	•	Mot de passe unique : Un seul mot de passe maître pour toute l'application
	•	Récupération : Envoi du mot de passe par email sur l'adresse email principale pré-enregistrée dans l'application

4. Structure de la base de données SQLite
Table champs_clients_actifs :
Objectif : Définir quels champs d'information clients sont activés pour cette entreprise
	•	id (INTEGER PRIMARY KEY AUTOINCREMENT)
	•	nom_champ (TEXT, ex: "date_naissance", "situation_maritale", "a_enfants", "a_parents", "parents_en_vie", etc.)
	•	actif (BOOLEAN, TRUE si le champ est utilisé)
	•	obligatoire (BOOLEAN, TRUE si le champ est obligatoire pour un profil complet)
	•	depend_de (TEXT, nom du champ parent, NULL si indépendant)
	•	valeur_dependance (TEXT, valeur que le champ parent doit avoir pour que ce champ soit affiché)
	•	type_champ (TEXT : "texte", "date", "nombre", "booleen", "liste", "email", "telephone")
	•	ordre_affichage (INTEGER, pour l'ordre dans le formulaire)
Champs par défaut disponibles (tous désactivables sauf nom et prénom) :
	•	nom (obligatoire, toujours actif)
	•	prenom (obligatoire, toujours actif)
	•	date_naissance (recommandé)
	•	adresse
	•	ville
	•	code_postal
	•	email
	•	telephone
	•	situation_maritale (liste : Célibataire / En couple / Marié(e) / Pacsé(e) / Divorcé(e) / Veuf/Veuve)
	•	a_conjoint (booléen, dépend de situation_maritale = "En couple" / "Marié(e)" / "Pacsé(e)")
	•	nom_conjoint (dépend de a_conjoint = TRUE)
	•	prenom_conjoint (dépend de a_conjoint = TRUE)
	•	date_naissance_conjoint (dépend de a_conjoint = TRUE)
	•	email_conjoint (dépend de a_conjoint = TRUE)
	•	telephone_conjoint (dépend de a_conjoint = TRUE)
	•	a_enfants (booléen)
	•	nombre_enfants (nombre, dépend de a_enfants = TRUE)
	•	a_parents (booléen)
	•	parents_en_vie (booléen, dépend de a_parents = TRUE)
	•	nom_pere (dépend de parents_en_vie = TRUE)
	•	prenom_pere (dépend de parents_en_vie = TRUE)
	•	date_naissance_pere (dépend de parents_en_vie = TRUE)
	•	email_pere (dépend de parents_en_vie = TRUE)
	•	telephone_pere (dépend de parents_en_vie = TRUE)
	•	nom_mere (dépend de parents_en_vie = TRUE)
	•	prenom_mere (dépend de parents_en_vie = TRUE)
	•	date_naissance_mere (dépend de parents_en_vie = TRUE)
	•	email_mere (dépend de parents_en_vie = TRUE)
	•	telephone_mere (dépend de parents_en_vie = TRUE)
	•	profession
	•	centre_interet
	•	notes_personnalisees (champ texte libre)
Table clients :
	•	id (INTEGER PRIMARY KEY AUTOINCREMENT)
	•	nom (TEXT, obligatoire, stocké en MAJUSCULES automatiquement)
	•	prenom (TEXT, obligatoire, première lettre en majuscule automatiquement)
	•	date_naissance (TEXT format JJ/MM/AAAA, optionnel mais recommandé)
	•	age (INTEGER, calculé automatiquement à partir de la date de naissance)
	•	adresse (TEXT, optionnel)
	•	ville (TEXT, optionnel)
	•	code_postal (TEXT, optionnel)
	•	email (TEXT, optionnel, validation format email)
	•	telephone (TEXT, optionnel)
	•	situation_maritale (TEXT, optionnel)
	•	a_conjoint (BOOLEAN, optionnel)
	•	a_enfants (BOOLEAN, optionnel)
	•	nombre_enfants (INTEGER, optionnel)
	•	a_parents (BOOLEAN, optionnel)
	•	parents_en_vie (BOOLEAN, optionnel)
	•	profession (TEXT, optionnel)
	•	centre_interet (TEXT, optionnel)
	•	notes_personnalisees (TEXT, optionnel)
	•	date_ajout (TIMESTAMP, automatique lors de la création)
	•	date_creation (TIMESTAMP, automatique)
	•	date_modification (TIMESTAMP, automatique)
	•	profil_complet (BOOLEAN, calculé automatiquement : TRUE si tous les champs obligatoires actifs sont renseignés)
	•	est_conjoint_de (INTEGER FOREIGN KEY → clients.id, NULL si client principal)
	•	est_enfant_de (INTEGER FOREIGN KEY → clients.id, NULL si client principal)
	•	est_parent_de (INTEGER FOREIGN KEY → clients.id, NULL si client principal)
	•	type_relation (TEXT : "principal", "conjoint", "enfant", "pere", "mere")
Table conjoints (simplification, liée à clients) :
	•	id (INTEGER PRIMARY KEY AUTOINCREMENT)
	•	client_id (INTEGER FOREIGN KEY → clients.id)
	•	conjoint_client_id (INTEGER FOREIGN KEY → clients.id, référence au compte client du conjoint)
Table enfants :
	•	id (INTEGER PRIMARY KEY AUTOINCREMENT)
	•	client_id (INTEGER FOREIGN KEY → clients.id, parent)
	•	enfant_client_id (INTEGER FOREIGN KEY → clients.id, référence au compte client de l'enfant)
	•	ordre (INTEGER, 1 pour enfant 1, 2 pour enfant 2, etc.)
Table parents_clients :
	•	id (INTEGER PRIMARY KEY AUTOINCREMENT)
	•	client_id (INTEGER FOREIGN KEY → clients.id)
	•	parent_client_id (INTEGER FOREIGN KEY → clients.id)
	•	type_parent (TEXT : "pere", "mere")
Table categories_produits :
	•	id (INTEGER PRIMARY KEY AUTOINCREMENT)
	•	nom (TEXT, défini par l'utilisateur)
	•	date_creation (TIMESTAMP)
Table attributs_produits :
	•	id (INTEGER PRIMARY KEY AUTOINCREMENT)
	•	categorie_id (INTEGER FOREIGN KEY → categories_produits.id)
	•	nom_attribut (TEXT, ex: "Couleur", "Matière", "Motif", "Marque", "Pays d'origine")
	•	type_attribut (TEXT : "texte", "nombre", "liste")
	•	par_defaut (BOOLEAN, TRUE pour "Marque" et autres attributs par défaut)
	•	ordre_affichage (INTEGER)
Attributs par défaut créés automatiquement pour chaque catégorie :
	•	Marque (texte, par défaut actif, supprimable/éditable)
	•	Pays d'origine (texte, par défaut actif, supprimable/éditable)
Table produits :
	•	id (INTEGER PRIMARY KEY AUTOINCREMENT)
	•	categorie_id (INTEGER FOREIGN KEY → categories_produits.id)
	•	nom (TEXT)
	•	prix (REAL, en euros par défaut)
	•	description (TEXT, optionnel)
	•	date_creation (TIMESTAMP)
Table valeurs_attributs_produits :
	•	id (INTEGER PRIMARY KEY AUTOINCREMENT)
	•	produit_id (INTEGER FOREIGN KEY → produits.id)
	•	attribut_id (INTEGER FOREIGN KEY → attributs_produits.id)
	•	valeur (TEXT)
Table ventes :
	•	id (INTEGER PRIMARY KEY AUTOINCREMENT)
	•	client_id (INTEGER FOREIGN KEY → clients.id)
	•	produit_id (INTEGER FOREIGN KEY → produits.id)
	•	quantite (INTEGER)
	•	prix_unitaire (REAL)
	•	prix_total (REAL, calculé automatiquement)
	•	date_vente (TIMESTAMP)
	•	notes (TEXT, optionnel)
Table mails_enregistres :
	•	id (INTEGER PRIMARY KEY AUTOINCREMENT)
	•	nom_mail (TEXT, titre du mail pour l'identifier)
	•	objet (TEXT)
	•	contenu_html (TEXT, contenu formaté en HTML)
	•	contenu_texte (TEXT, version texte brut)
	•	pieces_jointes (TEXT, chemins des fichiers séparés par ";")
	•	date_creation (TIMESTAMP)
	•	date_modification (TIMESTAMP)
Table historique_emails :
	•	id (INTEGER PRIMARY KEY AUTOINCREMENT)
	•	mail_id (INTEGER FOREIGN KEY → mails_enregistres.id, peut être NULL si mail non enregistré)
	•	objet (TEXT)
	•	contenu (TEXT)
	•	type_envoi (TEXT : "individuel", "selection", "tous", "reponse")
	•	nombre_destinataires (INTEGER)
	•	destinataires (TEXT, liste des emails séparés par ";")
	•	en_reponse_a (INTEGER FOREIGN KEY → emails_recus.id, NULL si pas une réponse)
	•	date_envoi (TIMESTAMP)
	•	statut (TEXT : "envoyé", "échoué", "partiel")
	•	compte_email_utilise (TEXT, adresse email utilisée pour l'envoi)
	•	erreurs (TEXT, détails des erreurs si échec)
Table emails_programmes :
	•	id (INTEGER PRIMARY KEY AUTOINCREMENT)
	•	mail_id (INTEGER FOREIGN KEY → mails_enregistres.id)
	•	type_envoi (TEXT : "individuel", "selection", "tous")
	•	filtres_selection (TEXT, JSON des filtres appliqués)
	•	date_programmation (TIMESTAMP, quand l'email doit être envoyé)
	•	heure_programmation (TEXT format HH:MM)
	•	date_creation (TIMESTAMP, quand la programmation a été créée)
	•	statut (TEXT : "en_attente", "envoyé", "annulé", "échoué")
Table emails_recus (nouvelle) :
Objectif : Stocker les emails reçus dans la boîte de réception centralisée
	•	id (INTEGER PRIMARY KEY AUTOINCREMENT)
	•	expediteur_email (TEXT, adresse email de l'expéditeur)
	•	expediteur_nom (TEXT, nom affiché de l'expéditeur)
	•	client_id (INTEGER FOREIGN KEY → clients.id, NULL si expéditeur non dans la base)
	•	objet (TEXT)
	•	contenu_html (TEXT)
	•	contenu_texte (TEXT)
	•	date_reception (TIMESTAMP)
	•	lu (BOOLEAN, FALSE par défaut)
	•	pieces_jointes (TEXT, chemins des fichiers téléchargés séparés par ";")
	•	compte_email_recepteur (TEXT, compte email sur lequel le mail a été reçu)
	•	message_id (TEXT, ID unique du message IMAP)
Table comptes_email :
	•	id (INTEGER PRIMARY KEY AUTOINCREMENT)
	•	adresse_email (TEXT)
	•	mot_de_passe_app (TEXT, crypté)
	•	nom_affichage (TEXT, nom affiché comme expéditeur)
	•	quota_journalier (INTEGER, défaut 500)
	•	quota_utilise_aujourd_hui (INTEGER, défaut 0)
	•	date_dernier_reset (DATE, pour réinitialiser le quota quotidien)
	•	actif (BOOLEAN, TRUE si le compte est utilisable)
	•	ordre_utilisation (INTEGER, pour définir l'ordre de rotation)
	•	serveur_smtp (TEXT, ex: "smtp.gmail.com")
	•	port_smtp (INTEGER, défaut 587)
	•	serveur_imap (TEXT, ex: "imap.gmail.com")
	•	port_imap (INTEGER, défaut 993)
	•	activer_reception (BOOLEAN, TRUE si on veut recevoir les emails sur ce compte)
Table parametres :
	•	cle (TEXT PRIMARY KEY)
	•	valeur (TEXT)
	•	type (TEXT : "texte", "nombre", "booleen")
	•	description (TEXT)
Paramètres stockés :
	•	mot_de_passe_patron (hashé)
	•	email_recuperation (email pour récupérer le mot de passe)
	•	monnaie (défaut : "EUR", peut être "USD")
	•	symbole_monnaie (défaut : "€", peut être "$")
	•	nom_entreprise (optionnel)
	•	date_creation_entreprise (pour les stats "depuis la création")
	•	langue_interface (défaut : "fr")
	•	theme (défaut : "clair")
	•	derniere_verif_emails (TIMESTAMP, pour la synchronisation IMAP)

5. Description détaillée des onglets
Onglet 1 : Client (accessible en mode vendeur)
Objectif : Créer et modifier les fiches clients
Interface dynamique : Les champs affichés dépendent de la configuration dans les Paramètres (champs actifs)
Formulaire de saisie avec champs (affichés uniquement si actifs) :
Informations de base :
	•	Nom (obligatoire, auto-MAJUSCULES)
	•	Prénom (obligatoire, auto-Capitale première lettre)
	•	Date de naissance (format JJ/MM/AAAA, calendrier sélectionnable) - Jamais demander l'âge directement
	•	Âge (calculé automatiquement en temps réel à partir de la date de naissance, affiché en lecture seule)
	•	Adresse
	•	Ville
	•	Code postal
	•	Email (validation format)
	•	Téléphone
	•	Profession
	•	Centres d'intérêt
Situation familiale (si activé) :
	•	Situation maritale (liste déroulante : Célibataire / En couple / Marié(e) / Pacsé(e) / Divorcé(e) / Veuf/Veuve)
Section Conjoint (si situation_maritale = "En couple" / "Marié(e)" / "Pacsé(e)" ET champ actif) :
	•	Checkbox "A un conjoint"
	•	Si coché, affichage d'un sous-formulaire :
	•	Nom du conjoint
	•	Prénom du conjoint
	•	Date de naissance du conjoint
	•	Email du conjoint
	•	Téléphone du conjoint
	•	Checkbox "Créer un compte client pour le conjoint" : Si coché, crée automatiquement un compte client lié pour le conjoint avec toutes ces infos
Section Enfants (si champ actif) :
	•	Checkbox "A des enfants"
	•	Si coché :
	•	Champ "Nombre d'enfants" (numérique)
	•	Génération automatique de X sous-formulaires (1 par enfant) :
	•	Nom de l'enfant X
	•	Prénom de l'enfant X
	•	Date de naissance de l'enfant X
	•	Email de l'enfant X (optionnel)
	•	Téléphone de l'enfant X (optionnel)
	•	Checkbox "Créer un compte client pour l'enfant X" : Si coché, crée automatiquement un compte client lié pour l'enfant
Section Parents (si champ actif) :
	•	Checkbox "A des parents"
	•	Si coché :
	•	Checkbox "Parents en vie"
	•	Si coché, affichage de deux sous-formulaires :Père :
	•	Nom du père
	•	Prénom du père
	•	Date de naissance du père
	•	Email du père
	•	Téléphone du père
	•	Checkbox "Créer un compte client pour le père"
	•	Mère :
	•	Nom de la mère
	•	Prénom de la mère
	•	Date de naissance de la mère
	•	Email de la mère
	•	Téléphone de la mère
	•	Checkbox "Créer un compte client pour la mère"
Notes personnalisées :
	•	Champ texte libre pour notes
Validation des données :
	•	Email : vérification du format (xxx@xxx.xxx)
	•	Date : format JJ/MM/AAAA automatique
	•	Nom : conversion automatique en MAJUSCULES
	•	Prénom : première lettre automatiquement en majuscule
	•	Téléphone : formatage automatique
Indicateur de profil :
	•	Barre de progression ou badge indiquant si le profil est "Complet" ou "Incomplet"
	•	Complet = Tous les champs obligatoires actifs sont renseignés
Boutons :
	•	"Enregistrer" : Sauvegarde le client dans la base de données (et crée les comptes liés si checkboxes cochées)
	•	"Annuler" : Réinitialise le formulaire
	•	"Nouveau" : Vide le formulaire pour créer un nouveau client
Gestion des relations :
	•	Si un conjoint/enfant/parent est créé en tant que compte client, il est lié au client principal
	•	Ces comptes liés apparaissent dans les recherches et peuvent recevoir des emails
	•	Ils peuvent aussi être ciblés indépendamment (ex: email uniquement aux enfants de 10-20 ans)

Onglet 2 : Vente (accessible en mode vendeur)
Objectif : Enregistrer les achats des clients
Interface :
Section 1 : Sélection du client
	•	Barre de recherche avec autocomplétion (recherche par nom, prénom, email, téléphone)
	•	Affichage : Nom PRENOM - Email - Téléphone (avec mise en gras des termes recherchés)
Section 2 : Sélection du produit
	•	Liste déroulante des catégories de produits
	•	Si la catégorie n'existe pas : Bouton "+ Créer une nouvelle catégorie" (popup rapide pour créer la catégorie)
	•	Liste déroulante des produits (filtrés par catégorie)
	•	Affichage : Nom du produit - Prix - Attributs principaux
	•	Si le produit n'existe pas : Bouton "+ Créer un nouveau produit" (ouvre un formulaire rapide) :
	•	Nom du produit
	•	Prix
	•	Saisie rapide des attributs de la catégorie
	•	Bouton "Créer et sélectionner"
	•	Si un attribut/détail manque pour le produit sélectionné :
	•	Affichage d'un avertissement : "Certains attributs ne sont pas renseignés"
	•	Bouton "Compléter les détails" : Ouvre un mini-formulaire pour remplir les attributs manquants
	•	Possibilité de continuer sans compléter (enregistrement partiel)
Section 3 : Détails de la vente
	•	Quantité (saisie numérique, défaut = 1)
	•	Prix unitaire (pré-rempli, modifiable)
	•	Prix total (calculé automatiquement : quantité × prix unitaire)
	•	Date de vente (par défaut = aujourd'hui, modifiable via calendrier)
	•	Notes (champ texte libre, optionnel)
Boutons :
	•	"Enregistrer la vente"
	•	"Annuler"
	•	"+ Ajouter un autre produit" (pour vente multiple au même client, sans revalider)
Historique des ventes du client sélectionné :
	•	Tableau affichant les 5 dernières ventes du client (produit, date, montant)

Onglet 3 : Recherche (accessible en mode vendeur)
Objectif : Rechercher et consulter les profils clients
Interface :
Barre de recherche (en haut) :
	•	Recherche intelligente dans : nom, prénom, email, téléphone
	•	Autocomplétion en temps réel
	•	Chaque mot est traité indépendamment
	•	Liste déroulante avec résultats : NOM Prénom - Email - Téléphone
	•	Mise en gras des termes correspondants
Profil client (après sélection) :
Section informations personnelles :
	•	Affichage de tous les champs actifs renseignés pour ce client
	•	Nom, Prénom, Âge (calculé), Date de naissance
	•	Adresse, Ville, Code postal
	•	Email, Téléphone
	•	Situation maritale
	•	Si a un conjoint : Affichage des infos du conjoint avec lien vers son profil
	•	Si a des enfants : Liste des enfants avec prénoms, âges, et liens vers leurs profils (si comptes créés)
	•	Si a des parents : Affichage des infos des parents avec liens vers leurs profils
Section statistiques d'achat :
	•	Nombre total d'achats
	•	Montant total dépensé
	•	Produit le plus acheté
	•	Catégorie de produits préférée
	•	Pays d'origine le plus acheté (si attribut actif)
	•	Marque la plus achetée (si attribut actif)
	•	Date du dernier achat
	•	Date d'ajout dans la base (client depuis le X/XX/XXXX)
Section graphiques :
	•	Graphique 1 : Évolution des achats dans le temps (courbe)
	•	Graphique 2 : Répartition par catégorie de produits (camembert)
	•	Graphique 3 : Dépenses mensuelles (histogramme)
	•	Graphique 4 : Répartition par pays d'origine (si attribut actif)
	•	Graphique 5 : Répartition par marque (si attribut actif)
Filtres de période pour les graphiques :
	•	Sélecteur : "1 jour", "1 semaine", "1 mois", "1 année", "Depuis l'ajout du client", "Depuis la création de l'entreprise"
	•	Calendrier pour sélectionner une date/période précise
Boutons :
	•	"Modifier le profil" (ouvre l'onglet Client en mode édition)
	•	"Voir l'historique d'achat complet" (ouvre un tableau détaillé)
	•	"Envoyer un email à ce client" (redirige vers l'onglet Emailing avec le client pré-sélectionné)

Onglet 4 : Emailing (accessible uniquement en mode patron)
Objectif : Envoyer des emails personnalisés à un ou plusieurs clients
Interface :
Section 1 : Sélection des destinataires (3 boutons radio) :
	1	Radio "Un seul client" :
	•	Affiche : Barre de recherche avec autocomplétion
	•	Sélection du client via liste déroulante
	2	Radio "Sélection de clients" :
	•	Affiche : Panneau de filtres dynamiques (en fonction des champs actifs dans les paramètres) :
	3	Filtres de base :
	•	Genre (checkbox : Homme / Femme, si champ actif)
	•	Âge (curseur : de X à Y ans)
	•	Situation maritale (checkboxes multiples, si champ actif)
	•	Ville (liste déroulante multi-sélection, si champ actif)
	•	Date d'ajout (de date A à date B)
	4	Filtres familiaux (si champs actifs) :
	•	A un conjoint (checkbox)
	•	A des enfants (checkbox)
	•	Nombre d'enfants (curseur : de X à Y, si "A des enfants" coché)
	•	Âge des enfants (curseur : de X à Y ans, si "A des enfants" coché)
	•	A des parents (checkbox)
	•	Parents en vie (checkbox, si "A des parents" coché)

12. Points techniques importants
Gestion des dates :
	•	Format uniforme : JJ/MM/AAAA
	•	Stockage en base : TEXT (SQLite) ou TIMESTAMP selon le cas
	•	Calcul automatique de l'âge : Basé sur la date de naissance actuelle (pas d'incrémentation arbitraire)
	•	Widget calendrier PySide6 (QDateEdit ou QCalendarWidget)
Éditeur de texte riche :
	•	Utilisation de QTextEdit avec QTextDocument (HTML)
	•	Conversion HTML → PDF avec reportlab
	•	Conversion HTML → DOCX avec python-docx
	•	Conversion HTML → ODT avec odfpy
Images dans les emails :
	•	Upload d'image → Conversion en base64
	•	Intégration dans le HTML : <img src="data:image/png;base64,...">
	•	Stockage dans la base de données si mail enregistré
Graphiques Plotly :
	•	Génération en HTML
	•	Intégration via QWebEngineView
	•	Export en image (PNG) avec kaleido
Envoi d'emails SMTP :
	•	Configuration TLS/SSL
	•	Port 587 (TLS) pour Gmail
	•	Gestion des erreurs (timeout, authentification, etc.)
	•	Retry en cas d'échec temporaire
	•	Rotation automatique des comptes selon les quotas
Réception d'emails IMAP :
	•	Connexion IMAP SSL (port 993 pour Gmail)
	•	Synchronisation automatique toutes les 5 minutes
	•	Téléchargement des pièces jointes
	•	Parsing du HTML et texte
	•	Matching automatique avec la base de données clients
Service d'envoi automatisé 24h/24 :
	•	APScheduler : Scheduler Python qui tourne en permanence
	•	Alternative : Daemon/Service système (launchd, systemd, Windows Service)
	•	Vérification toutes les minutes des emails programmés
	•	Envoi automatique à l'heure exacte
	•	Mise à jour de la base de données
	•	Fonctionne en arrière-plan même si l'app est fermée
Moteur de variables dynamiques :
	•	Détection des variables #NOM, #PRENOM, etc.
	•	Liste des variables disponibles générée dynamiquement selon :
	•	Les champs actifs dans les paramètres
	•	Les filtres de sélection appliqués
	•	Le type d'envoi (individuel / sélection / tous)
	•	Remplacement automatique lors de l'envoi
	•	Gestion des variables manquantes (remplacement par une chaîne vide ou valeur par défaut)
Système de champs clients personnalisables :
	•	Configuration flexible selon le type d'activité
